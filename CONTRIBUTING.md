# Contributing to openreversefeed

Thanks for looking. This is an early-alpha project so contributions are
welcome. This document describes the development workflow and the
public-safety rules that apply to every change.

## Ground rules

1. **No real financial reference data.** No real investor PAN numbers, no
   real ISIN codes, no real AMC or fund names. Sample generators use the
   `AAAPL*` PAN prefix and `SYN*` scheme-code prefix. The CI job
   `tools/check_forbidden_strings.py` will fail your PR if you add any.
2. **No internal identifiers.** No company names, internal domains,
   personal emails, or sibling-repo paths. The guard checks for all of
   them.
3. **The library must work when embedded in someone else's Django or
   FastAPI app.** That means no global state except `default_registry`,
   no hardcoded database URLs, and no imports from the reference app
   back into `src/openreversefeed/`.

## Development setup

```bash
git clone git@github.com:AngelOneWealth/openreversefeed.git
cd openreversefeed
python3.11 -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"

# bring up postgres
docker compose up -d postgres

# run the full unit suite
pytest tests/unit -v

# run the integration suite (needs docker + testcontainers)
pytest tests/integration -v -m integration

# run the forbidden-strings guard locally before pushing
python tools/check_forbidden_strings.py
```

## The Django reference app

The reference app under `examples/django_reference/` is a **demo** that
consumes the library via a SQLAlchemy bridge. It is not intended for
production. To run it:

```bash
cd examples/django_reference
python manage.py migrate --run-syncdb
python manage.py ofr_seed
python manage.py runserver 8765
```

Open http://127.0.0.1:8765. Upload a sample file generated by
`python tools/generate_samples.py`.

## Commit style

Short lowercase subject lines, one sentence. Body is optional and only
added when the change is non-obvious. No `feat(x):` or `fix:` prefixes.
No bullet lists inside commit bodies. Look at recent `git log --oneline`
for the tone.

## Pull requests

- Pass `ruff check src tests`.
- Add tests that fail before your change and pass after.
- Run `python tools/check_forbidden_strings.py` before opening the PR.
- Describe *why* in the PR body, not what — the diff shows what.

## Scope of contributions we're looking for

- New registrar adapters (any Indian AMC / RTA)
- Additional file format support (Excel variants, legacy DBF codepages)
- Tests that cover real-world edge cases the existing suite misses
- Performance improvements to the cleaner pipeline
- Documentation

We are **not** currently accepting:

- UI redesigns of the reference app — it's scaffolding
- New unrelated features that expand the library's scope
- Dependency changes without a clear reason
